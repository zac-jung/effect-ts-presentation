# 9. 정리와 결론

## 섹션 목표

Effect.ts를 좋은 아키텍처의 관점에서 다시 요약하고, 팀에게 남길 메시지를 정리한다.

---

## Slide 1. Effect.ts로 배우는 좋은 아키텍처

Effect.ts가 보여주는 좋은 아키텍처의 원칙:

```txt
1. 순수 로직과 Effect를 구분한다
2. Effect를 즉시 실행하지 않고 값으로 다룬다
3. 성공, 실패, 의존성을 타입에 드러낸다
4. 코드는 capability에 의존하고 runtime은 나중에 선택한다
5. 시간의 흐름은 runtime이 일관되게 제어한다
6. generator는 program과 runtime을 연결한다
```

---

## Slide 2. 처음 질문으로 돌아가기

좋은 아키텍처가 분리하려던 것:

```txt
1. 무엇을 할 것인가
2. 어떻게 실행할 것인가
3. 어떤 구현체를 사용할 것인가
4. 어떤 실패가 가능한가
5. 시간의 흐름을 누가 제어할 것인가
```

Effect.ts의 언어로 보면:

```txt
Effect<Success, Error, Requirements>
Runtime / Layer
Generator / Fiber
```

단, 발표에서는 이것을 1:1 치환표처럼 다루기보다 다음 메시지로 정리한다.

> Effect.ts는 좋은 아키텍처가 분리하려는 것들을 TypeScript 코드 안에서 명시적으로 드러내는 모델이다.

---

## Slide 3. 테스트 가능성으로 다시 보기

테스트하기 쉬운 코드는 대체로 다음을 만족한다.

```txt
- 의존성이 명시적이다
- 외부 환경을 교체할 수 있다
- 실패 상황을 재현할 수 있다
- 시간 흐름을 제어할 수 있다
- 순수 로직과 Effect가 분리되어 있다
```

Effect.ts에서는:

```txt
Program은 그대로 두고,
Runtime만 바꿔서 원하는 세계를 만든다.
```

### 핵심 문장

> 테스트 가능하다는 것은 같은 프로그램을 다른 runtime 위에서 실행할 수 있다는 뜻이다.

---

## Slide 4. Effect.ts가 아닌 것

Effect.ts는 다음이 아니다.

```txt
- 좋은 아키텍처를 자동으로 만들어주는 도구
- 모든 코드에 반드시 써야 하는 도구
- React component 로직을 전부 대체하는 도구
- 함수형 용어를 많이 알아야만 쓸 수 있는 도구
```

Effect.ts는 다음에 가깝다.

```txt
- Effect를 값으로 모델링하는 도구
- 실패와 의존성을 타입에 드러내는 도구
- code와 runtime을 분리하는 도구
- 비동기/시간/외부환경을 일관되게 다루는 runtime 모델
```

---

## Slide 5. 최종 메시지

> Effect.ts는 좋은 아키텍처를 대신 만들어주는 도구는 아니다.  
> 하지만 좋은 아키텍처가 요구하는 분리 — 순수 로직과 Effect, code와 runtime, success와 error, interface와 implementation — 를 TypeScript 코드 안에서 명시적으로 표현하게 해준다.

마지막 한 문장:

> Effect.ts를 배운다는 것은 단순히 라이브러리를 배우는 것이 아니라, 프론트엔드 코드에서 “무엇을 선언하고, 무엇을 runtime에게 맡길 것인가”를 다시 생각하는 것이다.

---

## 발표 이후 논의거리

- 우리 코드에서 순수 로직과 Effect가 섞여 있는 대표적인 곳은 어디인가?
- 테스트하기 어려운 시나리오는 대부분 어떤 Effect 때문인가?
- 어떤 영역부터 Effect.ts를 실험해볼 수 있을까?
- Effect.ts를 쓰지 않더라도 이 관점으로 개선할 수 있는 부분은 무엇인가?
