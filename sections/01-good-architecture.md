# 1. 좋은 아키텍처란 무엇인가

## 섹션 목표

Effect.ts 이야기를 시작하기 전에, 발표의 기준점이 되는 “좋은 아키텍처”를 정의한다.

## 핵심 메시지

> 좋은 아키텍처는 폴더 구조나 특정 패턴 이름이 아니라, 복잡한 소프트웨어를 오래 안전하게 변경하기 위한 구조다.

## Slide 1. 오늘의 주제

### 제목

Effect.ts를 배우기 전에, 좋은 아키텍처를 다시 생각해보자

### 내용

이 발표는 다음이 아니다.

- Effect.ts API 튜토리얼
- 함수형 프로그래밍 이론 강의
- 특정 코드베이스와 Effect.ts의 1:1 비교

이 발표는 다음에 가깝다.

- 우리가 이미 하던 best practice가 어떤 문제를 해결하려는지 환기
- Effect라는 고수준 개념 소개
- code와 runtime의 분리 이해
- generator가 왜 필요한지 이해

### 발표자 노트

Effect.ts를 “새로운 라이브러리”로 소개하면 러닝 커브가 먼저 보인다.  
반대로 좋은 아키텍처를 이해하는 렌즈로 소개하면, 팀이 이미 경험한 문제들과 자연스럽게 연결할 수 있다.

---

## Slide 2. 좋은 아키텍처는 폴더 구조가 아니다

### 내용

좋은 아키텍처가 만족해야 하는 것들:

```txt
- 변경에 강한가
- 책임이 잘 분리되어 있는가
- 구현체와 정책이 느슨하게 결합되어 있는가
- 외부 환경에 과하게 묶이지 않는가
- 실패를 명시적으로 다루는가
- 이해하고 추론하기 쉬운가
- 테스트하기 쉬운가
```

### 핵심 문장

> 좋은 아키텍처는 복잡성을 없애는 것이 아니라, 복잡성을 적절한 위치로 옮기는 일이다.

---

## Slide 3. 좋은 아키텍처가 분리하려는 것

좋은 아키텍처는 보통 다음을 분리하려고 한다.

```txt
1. 무엇을 할 것인가
2. 어떻게 실행할 것인가
3. 어떤 구현체를 사용할 것인가
4. 어떤 실패가 가능한가
5. 시간의 흐름을 누가 제어할 것인가
```

### 발표자 노트

이 다섯 질문이 이후 Effect.ts 설명의 골격이 된다.

- `Success`: 무엇을 만들 것인가
- `Error`: 어떤 실패가 가능한가
- `Requirements`: 어떤 capability가 필요한가
- `Runtime`: 어떻게 실행할 것인가
- `Generator`: 실행 제어권을 어떻게 runtime에게 넘길 것인가
